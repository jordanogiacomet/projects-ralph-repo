## Codebase Patterns
- Payload CMS 3.x is integrated into Next.js (not standalone) — config at `src/payload.config.ts`
- Frontend routes live in `src/app/(frontend)/`, Payload admin/API in `src/app/(payload)/`
- Use `@payload-config` path alias to import payload config
- Use `@/*` path alias for src imports
- Collections go in `src/collections/`, globals in `src/globals/`, blocks in `src/blocks/`
- Use `--legacy-peer-deps` for npm install due to Payload/Next.js peer dependency conflicts
- Next.js version must match Payload peer requirements (currently 15.4.x)
- Access control helpers live at `src/access/index.ts` — use `isAdmin`, `isAdminOrEditor`, `isLoggedIn`, `anyone`
- Slug auto-generation hook lives at `src/hooks/formatSlug.ts` — use `formatSlugHook` in field `hooks.beforeValidate`
- Collection slugs use kebab-case (for example `solucao-categories`, `cliente-segmentos`)
- Block definitions live in `src/blocks/`, exported via `src/blocks/index.ts`
- `BlockRenderer` at `src/components/BlockRenderer.tsx` maps `blockType` to React components
- Import blocks from the `@/blocks` barrel export when registering blocks in collections
- Interactive layout components use a server/client split: `FooServer.tsx` fetches data, `Foo.tsx` handles state/events
- Payload upload relations may resolve as string IDs or populated objects — always guard `typeof field === 'object'` before accessing nested properties such as `.url`
- When route slugs change, keep a backward-compatible route file that re-exports the canonical page using `export { default, generateMetadata } from '../canonical/page'`
- Social icons are inline SVGs in `SocialLinks.tsx` — no icon library is required
- Use `AnimatePresence` for conditional enter/exit animation surfaces
- Framer Motion surfaces should gate animation intensity with `useReducedMotion()` and disable drag/transition-heavy behavior for users with reduced-motion preferences
- Overlay-style UI (mobile menu, search) should trap focus with Tab/Shift+Tab and close on Escape while open
- Frontend typography is loaded via `next/font/google` in `src/app/(frontend)/layout.tsx`
- Global layout components should include deterministic fallback data when Payload globals are unavailable during static builds
- Footer defaults for unidades/social/cookie/WhatsApp are centralized in `src/lib/constants.ts` and merged with Payload globals in `Footer`
- Advanced contact forms follow a server-page + client-form split and submit JSON payloads to `/api/form-submissions`
- Until binary upload persistence is defined, file fields should store validated metadata in `data`
- Download-gated content flows can reuse `/api/form-submissions` with `formType: 'contato'` plus extra context fields in `data`
- Posts author relations may resolve as populated `users` objects or raw IDs, so UI mappers must guard `typeof author === 'object'`
- Route metadata should be authored via `buildMetadata` from `src/lib/seo.ts` to keep canonical, `hreflang`, Open Graph, Twitter Card, and icon tags consistent
- Public site origin resolution should prefer `NEXT_PUBLIC_SERVER_URL` and fall back to `NEXT_PUBLIC_SITE_URL`
- Structured data scripts are emitted with `JsonLd` in `src/components/JsonLd.tsx` and schema builders in `src/lib/seo.ts`
- Search discovery routes are centralized at `src/app/sitemap.ts` and `src/app/robots.ts`
- Global site search is served by `src/app/api/search/route.ts` and consumed by `SearchOverlay`; keep grouped responses by collection type (`pages`, `posts`, `solucoes`, `conteudos`)
- Payload CLI `run` imports scripts and exits when module import resolves, so async seed scripts should use top-level `await`
- Nested Payload array fields in Postgres should avoid reusing the same field name at consecutive levels (`children -> children`) to prevent relation-name collisions
- Phase-2 design tokens and aliases live in `src/styles/tokens.css` and `src/styles/utilities.css`
- Shared UI primitives live in `src/components/ui/` and should be preferred over one-off control styling when the same visual pattern appears in multiple templates
- Shared shell navigation normalization lives in `src/lib/navigation.ts` and should be reused anywhere header/footer navigation needs the same resilient Payload + fallback structure
- Redesign work should prefer front-end evolution over CMS schema changes unless there is clear editorial-governance value
- Payload-inspired visual language must be translated to Apollo’s institutional context; do not drift into a literal SaaS clone of Payload
- The header is functional but is not final merely because it works; current redesign backlog explicitly requires continued shell refinement

## Validation Rules
- Practical validation order in this repository:
  - `npm run lint`
  - `npx tsc --noEmit`
  - `npm run build`
- `npm run typecheck` is not defined unless explicitly added later
- `npm run test` is not defined unless explicitly added later
- In this repository, `npx tsc --noEmit` is the practical type-check gate
- In this repository, `npm run build` is the practical integration/build gate
- If `.next/types` is missing and `npx tsc --noEmit` fails because of that, rerun type validation after a successful Next build path
- If plain `npm run build` is flaky in the sandbox, `npx next build --debug` may be used as a diagnostic fallback, but the result must be logged precisely
- Manual browser QA is still required for UI stories when browser tooling is unavailable

## Active Risks / Known Issues
- Sandbox/static builds may log Postgres connectivity errors during static generation even when build exits successfully; distinguish runtime log noise from actual build failure
- Plain `npm run build` may intermittently fail in the sandbox with transient Next.js worker/module/page-resolution issues; record retries or `--debug` runs clearly
- Mobile and tablet experience must not be treated as a late adaptation pass during redesign work
- There is risk of mixing old phase-1 visual language and new phase-2 visual language for too long; redesign stories should aim for coherent template-level upgrades, not isolated decorative tweaks
- There is risk of drifting into a literal Payload clone; redesign decisions must preserve Apollo’s institutional tone and commercial structure
- Do not reopen CMS modeling just to solve purely visual problems unless editorial governance clearly improves

## Phase-2 Redesign Guardrails
- Modernize without decharacterizing Apollo
- Preserve institutional seriousness with more mature execution
- Increase whitespace and reduce visual noise
- Make cards, grids, filters, overlays, and forms feel more premium
- Use motion as refinement, not spectacle
- Keep the site consultive, technical, and commercially clear
- Keep SEO, accessibility, keyboard navigation, reduced motion, and route integrity intact
- Prefer design-system and template evolution over CMS remodeling
