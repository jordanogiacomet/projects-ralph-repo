## Codebase Patterns
- Project lives in the existing Apollo repository root; all commands must run from that root.
- This visual cycle does NOT replace the frozen baseline; it runs on top of it.
- The active source of truth for this round is `prd.json` in the worktree root.
- The baseline phase-2 history lives under `.ralph/baseline/`.
- This cycle is screenshot-driven and issue-driven: visual fixes must reference `docs/visual-register.md`.
- Use issue IDs in the format `VR-001`, `VR-002`, etc.
- Implement exactly one visual story per iteration.
- Within a story, fix only issues that belong to that story’s scope.
- Do not reopen broad redesign work when the issue is a localized spacing/alignment/polish problem.
- Prefer the smallest safe fix in the nearest responsible component.
- Reuse existing design-system primitives and shared utilities before introducing new one-off styling.
- If a problem exists in multiple routes because of one shared component, fix the shared component once and record all affected routes.
- Every completed story must update:
  - `prd.json`
  - `progress.txt`
  - `docs/visual-register.md`
- If checks fail, do not mark the story complete.
- If browser tooling is unavailable, explicitly record manual QA as pending.
- Do not rewrite the legacy `progress-phase2.txt` from the old cycle during this round.
