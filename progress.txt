## Codebase Patterns
- Payload CMS 3.x is integrated into Next.js (not standalone) - config at `src/payload.config.ts`
- Frontend routes live in `src/app/(frontend)/`, Payload admin/API in `src/app/(payload)/`
- Use `@payload-config` for the Payload config alias and `@/*` for `src` imports
- Collections live in `src/collections/`, globals in `src/globals/`, and blocks in `src/blocks/`
- Access helpers live in `src/access/index.ts` - prefer `isAdmin`, `isAdminOrEditor`, `isLoggedIn`, and `anyone`
- Slug normalization should reuse `formatSlugHook` from `src/hooks/formatSlug.ts` in `hooks.beforeValidate`
- Block definitions must be exported from `src/blocks/index.ts` and mapped through `src/components/BlockRenderer.tsx`
- Interactive layout surfaces follow a server/client split: server wrappers fetch data, client components own state and events
- Payload upload and relation fields may resolve as string IDs or populated objects - guard `typeof field === "object"` before nested access
- When route slugs change, preserve compatibility with a re-export route: `export { default, generateMetadata } from "../canonical/page"`
- Global shell components need deterministic fallback data so static builds still render when Payload or Postgres is unavailable
- Footer fallback values for unidades, social links, cookie copy, and WhatsApp live in `src/lib/constants.ts` and are merged with Payload globals
- Public contact/download flows reuse `/api/form-submissions`; advanced forms keep a server-page plus client-form split
- Shared public-form feedback and summary surfaces live in `src/components/PublicFormStatus.tsx` and `src/components/PublicFormSummary.tsx`; reuse them across contact/newsletter flows before introducing one-off submit-state UI
- Posts author relations may resolve as populated `users` objects or raw IDs, so UI mappers must guard `typeof author === "object"`
- Route metadata should be authored with `buildMetadata` from `src/lib/seo.ts`; structured data is emitted with `JsonLd`
- Public origin resolution should prefer `NEXT_PUBLIC_SERVER_URL` and fall back to `NEXT_PUBLIC_SITE_URL`
- Search discovery routes live in `src/app/sitemap.ts` and `src/app/robots.ts`; site search is served by `src/app/api/search/route.ts`
- Search responses stay grouped by collection type (`pages`, `posts`, `solucoes`, `conteudos`) so overlays can render resilient sectioned results
- Section spacing should use the shared utilities from `src/styles/utilities.css`: `section-space-tight`, `section-space`, and `section-space-loose`
- Hero sections use the shared asymmetric padding rhythm `pb-24 pt-28 sm:pb-28 sm:pt-32 lg:pb-32 lg:pt-36`
- Overlap sections should use `-mt-12 sm:-mt-16` with `pb-[var(--space-section-loose)]` to stay aligned with the redesign system
- Shared motion behavior belongs in `src/styles/tokens.css` and `src/styles/utilities.css`; prefer `motion-transition`, `motion-lift-*`, and `motion-media-scale` utilities
- Shared UI primitives in `src/components/ui/` should be preferred over one-off control styling
- Dark-surface secondary CTAs should use `Button` variant `inverted` from `src/components/ui/Button.tsx` instead of duplicating white-glass button classes per template
- Shared shell-navigation normalization lives in `src/lib/navigation.ts` and should be reused by header/footer navigation loaders
- Floating shell controls coordinate through `body[data-cookie-consent-visible]` plus the shared offset variables in `src/styles/utilities.css`; keep cookie, WhatsApp, and scroll-to-top stacking aligned there
- Canonical solution detail routes use per-page config plus the shared `src/components/SolutionDetailPage.tsx` template/loader; keep alias routes when historical slugs must resolve
- Solution category navigation counts should derive from the deterministic `orderedSolutionSlugs` config in `src/lib/solutionCategories.ts` so hubs stay stable even when Payload data is unavailable
- When premium route media is optional, prefer layered gradient fallback treatments in-component instead of hard-coded placeholder image URLs that can 404
- With multiple top-level route groups acting as root layouts, branded public 404s should live in segment-level `not-found.tsx` files (for example `news/not-found.tsx`) rather than a root `src/app/not-found.tsx`

## Validation Rules
- Required validation order for this repository:
  - `npm run lint`
  - `npm run typecheck` if it exists; otherwise `npx tsc --noEmit`
  - `npm run test` if it exists
  - `npm run build`
- `npm run typecheck` is not currently defined in `package.json`, so `npx tsc --noEmit` is the practical type-check gate.
- `npm run test` is not currently defined in `package.json`.
- If `.next/types` is missing and `npx tsc --noEmit` fails because of that, rerun type validation after a successful Next build path.
- If `npm run build` fails with known transient Next.js sandbox issues, retry once and record the exact result.
- If `npx next build --debug` is used for diagnosis, log it explicitly and do not silently treat it as the same as `npm run build`.
- Browser verification is required only for UI stories; when tooling is unavailable, record that manual QA is still pending.
